## RTK 省 Token 规则（本仓库）

- 目标：在不影响正确性的前提下，优先减少上下文长度与命令输出体积。
- 统一习惯：能用 `rtk <command>` 就不用裸命令。
- 若命令支持超紧凑模式，优先使用 `-u` 或 `--ultra`。
- 避免一次性读取大文件全文；优先按需读取（关键片段、关键函数、关键行号）。
- 搜索优先 `rg`，并限制范围到相关目录，避免全仓库无界扫描。

## 命令执行约定

- 文件发现：`rtk rg --files app web tests scripts`
- 文本搜索：`rtk rg "<pattern>" app web tests`
- 先看结构再下钻：先 `rg` 定位，再读取小片段，不直接整文件 `cat`。
- 测试执行：先跑最小相关测试，再决定是否扩大范围。
- 日志处理：只保留关键报错与关键信号，避免粘贴大段无关输出。

## 上下文控制策略

- 一次只处理一个明确子问题；避免并行展开多个无关方向。
- 回答默认“结论 + 关键依据 + 下一步”，避免长篇复述过程。
- 除非用户明确要求，不重复展示已看过的大段内容。
- 涉及前端改动时，只读取相关视图/组件文件，不扫全部 `web/frontend/src`。
- 涉及后端改动时，优先定位 `web/backend` 与 `app` 相关模块，避免跨层过度读取。

## 本仓库优先关注目录

- 后端主逻辑：`app/`、`web/backend/`
- 前端主逻辑：`web/frontend/src/`
- 测试：`tests/`
- 脚本：`scripts/`

## 高成本目录保护

- 非必要不读取：`data/backups/`、大体积 `data/eval/` 历史文件、`web/frontend/package-lock.json`
- 非必要不扫描：`.git/`、`__pycache__/`

## 会话内检查点

- 完成一轮改动后可执行 `rtk gain` 查看节省效果。
- 若节省不明显，优先收紧：
  1. 搜索范围
  2. 读取片段长度
  3. 输出摘要长度
